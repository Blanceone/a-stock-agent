## 1. 系统总览

### 1.1 业务定位

本系统是一个 **A股宏观锚定投研智能体**，核心理念：

> 以国家政策为宏观锚 → 拆解产业链 → 映射实体股票 → 动态监控消息面共振信号 → 输出预警

### 1.2 两条核心流水线

| 流水线 | 触发方式 | 输入 | 输出 | 图编排 |
|--------|---------|------|------|--------|
| **静态图谱** | 用户提供政策 PDF | 政策文件 | 第一/第二梯队候选股 | `static_graph.py` |
| **动态监控** | 常驻运行 | 实时新闻流 | 三共振预警信号 | `dynamic_graph.py` |

### 1.3 技术栈

| 层 | 技术选型 |
|----|---------|
| LLM | DeepSeek 双模型（deepseek-chat 粗筛/flash，deepseek-reasoner 深度推理/pro） |
| 图编排 | LangGraph StateGraph |
| 关系数据库 | PostgreSQL（pg_pool 连接池） |
| 向量数据库 | ChromaDB（bge-small-zh-v1.5 嵌入模型） |
| 缓存 | Redis（LLM 缓存、去重、新闻 feed、概念库） |
| Web | FastAPI + 单页 HTML 仪表盘 |
| 数据源 | Tushare Pro / AKShare / 东方财富直连 API / 财联社直连 API |
| 搜索 | SearXNG（私有部署，限流 10 次/分钟） |

## 3. 基础设施层 (infrastructure)

### 3.1 database.py — 统一数据连接管理

**业务职责**：统一管理 PostgreSQL / ChromaDB / Redis 三大数据源的连接生命周期。

**设计要点**：

- **PostgreSQL**：`SimpleConnectionPool(minconn=1, maxconn=10)`，`get_pg_conn()` / `release_pg_conn()` 模式，3 次自动重连。建表 DDL 在初始化时自动执行。
- **ChromaDB**：双版本兼容设计，优先探测 v0.x REST API（`/api/v1/heartbeat`），降级到 v1.x HttpClient。`_ChromaDBCompat` 和 `_CollectionCompat` 包装器实现透明 REST 通信。
- **Redis**：`decode_responses=True`，`ensure_redis()` 提供自动重连能力。
- **降级策略**：ChromaDB 不可用不崩溃（语义搜索降级），Redis 不可用不崩溃（缓存/去重降级）。

**PG 表结构**（5 张表）：

| 表名 | 职责 | 主键 |
|------|------|------|
| `stock_basic` | 全A股基础信息 | ts_code |
| `news_analysis` | 新闻分析结果 | article_id |
| `concept_stocks` | 概念→股票映射 | (concept, ts_code) |
| `sop_pending` | 待学习 SOP | id (SERIAL) |
| `sop_active` | 已提取待审核 SOP | id (SERIAL) |

**Redis Key 规范**：

| Key 模式 | 类型 | TTL | 职责 |
|---------|------|-----|------|
| `dedup:news:{article_id}` | String | 7d | 新闻去重 |
| `dedup:url:{url_md5}` | String | 7d | URL 内容缓存 |
| `llm:{prompt_hash}` | String | 24h | LLM 响应缓存 |
| `rate:searxng:{minute}` | String | 120s | SearXNG 限流计数器 |
| `dynamic:news_feed` | List | 7d | 新闻 Feed（LPush，最多 500 条） |
| `dynamic:news_analysis` | Hash | 7d | 新闻分析结果（article_id → JSON） |
| `dynamic:concepts` | Hash | 7d | 概念词库（concept → JSON） |
| `dynamic:alerts:{date}` | List | 7d | 当日预警信号 |
| `dynamic:concept_news:{concept}` | ZSet | 7d | 概念→新闻反向映射 |
| `dynamic:concept_stock_scores:{concept}` | Hash | 7d | 概念→股票评分 |
| `static:stock_pool` | String | - | 静态图谱输出股池 |
| `cache:concept_llm_eval` | String | 7d | 概念 LLM 评估缓存 |
| `concept_graph:edges` | Hash | 30d | 概念图谱边（child→JSON{parents,relevance,created_at}） |
| `concept_graph:roots` | Set | 30d | 概念图谱 Layer0 根节点集合 |
| `concept_graph:layer:{depth}` | Set | 30d | 概念图谱每层概念名集合 |
| `concept_graph:build_progress` | String | 1h | 概念图谱构建进度（JSON） |
| `concept_graph:build_lock` | String | 5min | 概念图谱并发构建锁 |

### 3.2 data_fetcher.py — 结构化数据降级链

**业务职责**：行情/财务/主营业务文本的统一数据获取入口。

**降级链设计**：

| 数据类型 | L1（首选） | L2（降级） | 说明 |
|---------|-----------|-----------|------|
| 全A股基础信息 | Tushare `stock_basic` | 无 | 唯一权威源 |
| 日线行情 | Tushare `pro.daily` | 东方财富 push2his | 捕获异常后 fallback |
| 盘中主力净流入 | 东方财富 push2（直连） | 腾讯财经 qt.gtimg.cn | 5 次重试退避 2s/4s/6s/8s，失败降级腾讯外盘/内盘数据 |
| 龙虎榜 | Tushare `pro.top_list` | 无 | T+1 数据 |
| 主营业务文本 | 东方财富 F10 核心题材 | Tushare `stock_company` | 两级降级 |

**异常分类**：
- `TushareQuotaError`：积分不足
- `TushareRateLimitError`：接口超频
- `DataFetchError`：所有来源均失败

### 3.3 concept_sources.py — 多源概念采集 + LLM 智能筛选

**业务职责**：采集 A 股概念板块数据，通过 LLM 评估国家政策关联度并排序筛选。

**数据流**：
```
akshare 东财概念列表（~494个）
    ↓
LLM 分批评估（每批80个，Flash模型）
  → policy_score (0-10): 国家政策关联度
  → category: 产业分类
  → keep (true/false): 是否保留
    ↓
按 policy_score 降序排序，仅保留 keep=True 的概念
    ↓
仅为保留的概念获取成分股（akshare.stock_board_concept_cons_em）
    ↓
返回 [{"concept", "stocks", "source", "score", "category"}]
```

**缓存策略**：LLM 评估结果缓存到 Redis `cache:concept_llm_eval`（7天），避免重复调用。

**降级策略**：LLM 不可用时退化为前缀过滤（`昨日/今日/近端/连续/次新`等）+ 截断 150 个。

**tushare 概念**：需 5000+ 积分，当前 2000 积分不可用，静默跳过。

**三重来源补股机制**（`find_stocks_for_concept()`）：

对无成分股的概念逐级兜底获取股票，全部失败才判定为无效概念：

| 层级 | 来源 | 说明 |
|------|------|------|
| Tier 1 | akshare API（含3次重试） | `stock_board_concept_cons_em` 直接获取 |
| Tier 2 | ChromaDB 语义搜索 | `enrich_concept_stocks()` 基于主营业务文本匹配 |
| Tier 3 | Web 搜索 + LLM | `call_llm_with_tools(web_search)` 搜索A股股票并 LLM 筛选 |

**批量刷新功能**（`refresh_all_concept_stocks()`）：遍历 `dynamic:concepts` 中所有概念，对每个执行三重来源重新获取成分股。

**概念列表同步修复**：`sync_concepts_to_redis()` 不再因 akshare 返回空股票而跳过概念（移除 `if not stocks: continue`），确保语义增强有机会执行。

### 3.4 searxng_search.py — 搜索引擎封装

**业务职责**：封装 SearXNG 搜索 API，提供结构化搜索结果。

**红线**：仅供 `static_graph` 产业链拆解和 `sop_learner` 低频调用，**严禁在 `dynamic_graph` 高频轮询节点中直接调用**（LLM tool-call 中使用除外，因有 LLM 自主决策频率）。

**限流**：Redis 滑动窗口计数器，≤10 次/分钟（可配置）。

### 3.5 web_fetcher.py — 网页正文三层降级提取

**业务职责**：从 URL 提取网页正文，供产业链拆解使用。

| 层级 | 技术 | 适用场景 | 缓存 |
|------|------|---------|------|
| L1 | defuddle 微服务（Node.js） | 静态 HTML | Redis 7d |
| L2 | Playwright 无头浏览器 | JS 渲染页面 | Redis 7d |
| L3 | requests + BeautifulSoup | 最终兜底 | Redis 7d |

### 3.6 semantic_init.py — 全A股语义知识库初始化

**业务职责**：将全 A 股主营业务文本向量化存入 ChromaDB，并 LLM 分类概念。

**流程**：
```
PostgreSQL stock_basic（全A股，排除ST）
    ↓
fetch_business_description（四级降级链获取主营文本）
    ↓
V4-Flash 摘要（≤200字，去冗余）
    ↓
LLM 概念分类（从已知概念列表中选取匹配项 + 输出 confidence）
    ↓
ChromaDB 向量化存储（bge-small-zh-v1.5，cosine 相似度）
    ↓
Redis dynamic:concepts 持久化（source=llm_classify, confidence=LLM输出）
```

**增量更新**：跳过已存在于 ChromaDB 中的 ts_code。每 50 只批量写入，每 100 只输出进度。

### 3.7 news_sources/ — 多源新闻聚合

**业务职责**：三个独立新闻源并行轮询，统一去重，推入处理队列。

#### 3.7.1 base.py — 聚合器基类

**NewsItem 数据模型**：
```python
@dataclass
class NewsItem:
    article_id: str       # 唯一ID（带源前缀: cls:12345, gov:abc123, csrc:xxx）
    title: str
    summary: str
    pub_time: datetime
    source: str = "财联社"
    source_type: str = "telegraph"   # "telegraph" | "policy"
    url: str = ""
    extra: dict = field(default_factory=dict)
```

**NewsAggregator 核心机制**：
- 每源一个 `asyncio.Task`，独立间隔轮询
- Redis `dedup:news:{article_id}` 去重（TTL 7天）
- 新消息到达 → 写入 `dynamic:news_feed`（LPush，最多 500 条）→ 立即 `asyncio.create_task(self._run_process(item))`
- `asyncio.Semaphore(5)` 控制最大并发处理数

#### 3.7.2 三个新闻源

| 源 | 类名 | 轮询间隔 | API | SSL |
|----|------|---------|-----|-----|
| 财联社电报 | `CLSTelegraphSource` | 5s | `cls.cn/api/cache` | 标准 |
| 国务院政策 | `GovPolicySource` | 60s | `sousuo.www.gov.cn` | 跳过验证 |
| 证监会公告 | `CSRCSource` | 60s | `csrc.gov.cn/searchList` | 跳过验证 |

## 4. 节点层 (nodes)

### 4.1 llm_utils.py — LLM 调用基础设施

**业务职责**：封装 DeepSeek API 调用，提供缓存、JSON 解析、Tool-Call 能力。

**双模型策略**：

| 模型 | ID | 用途 | 典型场景 |
|------|-----|------|---------|
| Flash | `deepseek-chat` | 高频/粗筛/摘要/排序 | 新闻粗筛、概念评估、技术面排序 |
| Pro | `deepseek-reasoner` | 深度推理/拆解/精判 | 政策解析、产业链拆解、新闻深读 |

**核心函数**：

| 函数 | 职责 | 缓存 | 返回 |
|------|------|------|------|
| `call_llm()` | 基础 LLM 调用 | Redis 24h | `str` |
| `call_llm_json()` | LLM 调用 + JSON 解析 | Redis 24h | `dict` / `list` |
| `call_llm_with_tools()` | 带 Tool-Call 的 LLM 调用 | 不缓存 | `dict` / `list` |
| `load_prompt()` | 加载 prompt 模板 | - | `str` |

**Tool-Call 流程**：
```
LLM(user prompt)
  → tool_calls: [web_search("概念含义")]
  → _execute_tool("web_search", {query: "..."})
  → SearXNG 搜索 → 返回 JSON 结果
  → tool role message 回传
  → LLM(最终回答)
  → _parse_json(content)
  
最多 max_rounds=5 轮，防止死循环
``**JSON 鲁棒解析**（`_parse_json`）：
1. 去掉 markdown code fence（` ```json ... ``` `）
2. 直接 `json.loads`
3. 提取 `{...}` 或 `[...]` 子串再解析
4. 修复尾逗号 + 去控制字符后重试

### 4.2 policy_parser.py — 政策解析节点

**业务职责**：从政策 PDF 中提取产业概念词。

**流程**：
```
PDF 文件
  ↓ _extract_toc() — 提取目录
  ↓ V4-Pro 圈定重点章节（返回 page numbers）
  ↓ _extract_section_texts() — 提取章节文本
  ↓ _store_policy_chunks() — 向量化存入 ChromaDB
  ↓ V4-Pro 提取产业概念词（返回 [{concept, source_section, confidence}]）
  ↓ 过滤空 concept 字段（信任 prompt 已排除宏观空词）
  ↓
输出: concepts list
```

**LLM 驱动改造**：已删除 `BANNED_WORDS` 硬编码黑名单（12个宏观空词），完全信任 LLM prompt 中的过滤指令。代码仅过滤 `concept` 字段为空的条目。

### 4.3 chain_splitter.py — 产业链深度拆解

**业务职责**：为每个概念搜索研报并拆解产业链层级结构。

**流程**：
```
每个概念
  ↓ _search_reports_with_llm() — LLM tool-call 自主搜索研报 URL
    → LLM 决定搜索关键词 → SearXNG 搜索 → 选取最佳 URL
    → 降级: 固定模板 f"{concept} 产业链 研报"
  ↓ _fetch_report_texts() — 三层降级抓取正文（defuddle → Playwright → BS4）
  ↓ _split_chain() — V4-Pro 拆解产业链层级
    → 输出: {layers: [{layer_name, nodes: [{name, role}]}], source_urls}
  ↓
输出: industry_chains list
```

**LLM 驱动改造**：搜索关键词由 LLM 自主决定（通过 `call_llm_with_tools`），不再使用固定模板 `f"{concept} 产业链 研报"`。

### 4.4 entity_mapper.py — 实体映射节点

**业务职责**：将产业链节点映射到具体 A 股股票。

**流程**：
```
industry_chains（产业链层级结构）
  ↓ 遍历每个产业链的每个节点
  ↓ ChromaDB 语义检索 top_k=30（cosine 相似度）
  ↓ 过滤：circ_mv ≤ 500亿 + 非 ST
  ↓ V4-Flash 相关性打分（输出 keep: true/false）
  ↓ 仅保留 keep=true 的股票
  ↓ PostgreSQL stock_basic 补充 name/industry
  ↓
输出: stock_pool [{ts_code, name, circ_mv, concept, chain_role}]
```

**LLM 驱动改造**：
- 过滤条件从 `score >= 0.6` 改为 LLM 输出的 `keep` 字段
- 市值上限从 200 亿放宽到 500 亿
- 降级回退：`s.get("keep", s.get("score", 0) >= settings.entity_llm_score_threshold)`

### 4.5 news_funnel.py — 新闻漏斗节点

**业务职责**：对每条新闻进行两级筛选（粗筛→深读），输出结构化分析结果。

**流程**：
```
NewsItem（新闻条目）
  ↓
一级粗筛 V4-Flash:
  → is_relevant: true/false（是否与A股投资相关）
  → news_score: 0-1
  ↓ is_relevant=false → 返回 None（过滤）
  ↓
二级深读 V4-Pro:
  → news_score: 0-1（重新评分）
  → impact_type: 利好/利空/中性
  → impact_concept: 核心概念
  → sentiment: positive/negative/neutral
  → related_ts_codes: [关联股票代码]
  → new_concept_terms: [新概念词]
  → mentioned_companies: [{name, impact, reason}]
  → supply_chain_impact: [{name, relation, impact, reason}]
  ↓
更新概念词库（新概念入库）
  ↓
输出: news_result dict
```

**LLM 驱动改造**：粗筛判断从 `score < 0.4` 改为 LLM 输出的 `is_relevant` 字段。

### 4.6 resonance_alert.py — 多因子共振预警节点

**业务职责**：综合判断消息面+资金面+技术面的共振信号，输出预警。

**流程**：
```
news_result + concepts_updated
  ↓
_collect_stock_data():
  → 获取关联股票列表
  → 拉取日线行情（data_fetcher.fetch_daily）
  → 计算技术指标（indicator_calc.calc_all）
  → 获取盘中主力净流入（data_fetcher.fetch_moneyflow_intraday）
  → 查询间隔 2s（东财 push2 限流保护）
  → 返回数据含 source 字段标识数据源（eastmoney-push2 / tencent / error）
  ↓
_llm_judge() V4-Flash:
  → 输入: news_title, news_score, stocks_data（多因子指标）
  → LLM 综合判断每只股票是否预警
  → 判断原则: 不需要三因子全部达标，综合信号足够强即可
  → 输出: [{ts_code, alert: true/false, confidence, reason}]
  ↓
降级 _fallback_judge():
  → 三阈值 AND 逻辑:
    news_score ≥ 0.7 AND capital_inflow ≥ 2% AND volume_ratio ≥ 2.0
  ↓
输出: resonance_alerts list + 写入 Redis dynamic:alerts:{date}
```

**LLM 驱动改造**：从固定三阈值 AND 逻辑改为 LLM 综合判断。降级时仍使用三阈值 AND。

### 4.7 tech_ranker.py — 技术面动态排序节点

**业务职责**：对候选股票池进行多因子技术面排序，分为第一/第二梯队。

**流程**：
```
stock_pool（候选股票）
  ↓
_fetch_klines(): 拉取每只股票近 30 日 K 线
  ↓
calc_all()（indicator_calc 纯数值计算红线）:
  → ma_align_score: 均线多头排列得分
  → vol_ratio_score: 量比得分
  → gain_score: 近期涨幅得分
  → small_cap_score: 小盘溢价得分
  ↓
构建 stocks_indicators JSON
  ↓
_llm_rank() V4-Flash:
  → 输入: 所有股票的技术指标
  → LLM 动态决定权重（根据市场环境）
  → 输出: [{ts_code, score, tier: "tier1"/"tier2"/"drop", reason}]
  ↓
降级 _fallback_rank():
  → 固定权重公式: score = 0.40*ma + 0.25*vr + 0.20*gain + 0.15*cap
  → tier1: score ≥ 0.7, tier2: score ≥ 0.4
  ↓
输出: {tier1: [...], tier2: [...]}
```

**LLM 驱动改造**：从固定权重公式（0.40/0.25/0.20/0.15）改为 LLM 动态权重排序。降级时使用固定权重。

### 4.8 sop_learner.py — SOP 自学习节点

**业务职责**：从政策文本中提取 SOP 操作图谱，写入待审核表。

**流程**：
```
PostgreSQL sop_pending (status='pending')
  ↓
_extract_sop_graph() V4-Flash:
  → 结构化提取操作图谱（JSON Schema）
  ↓
_write_to_active():
  → 写入 sop_active 表（approved=FALSE）
  → 等待人工在 Web UI 审核
  ↓
标记 sop_pending.status = 'processed'
  ↓
输出: {sop_processed: count}
```

**红线**：只能写入 `sop_pending → sop_active` 中间表，不可直接修改 `system_policy` 图谱。

### 4.9 concept_graph_builder.py — 政策驱动概念图谱引擎

**业务职责**：以十五五政策为锚点，通过 BFS 迭代扩展 + LLM 智能收敛构建层级概念图谱，支持全量构建、手动输入和新闻增量插入三种触发模式。

**三种触发模式**：

| 模式 | 触发方式 | 入口函数 | 说明 |
|------|---------|---------|------|
| 全量构建 | Web按钮 / CLI `--mode concept_graph` | `build_full()` | 下载十五五→提取Layer0→BFS展开→智能收敛 |
| 手动输入 | Web输入框 | `expand_from_concept()` | 匹配已有→未找到则录入并展开 |
| 增量插入 | 新闻发现新概念自动触发 | `incremental_insert()` | 评估关联度→挂到合适位置→有限展开 |

**核心函数**：

| 函数 | 职责 | 模型 | 说明 |
|------|------|------|------|
| `fetch_policy_text()` | 获取政策文本 | Pro | 三层降级：用户提供→文件路径→联网搜索+web_fetcher |
| `extract_layer0()` | 提取Layer0核心概念 | Pro | 从政策文本提取10-20个核心概念（含policy_basis/importance/category） |
| `bfs_expand()` | BFS迭代扩展 | Pro+web_search | 每节点发现3-8个子概念，relevance>=threshold入队继续扩展 |
| `build_full()` | 全量构建入口 | - | 含并发锁（5min TTL），按层BFS展开 |
| `expand_from_concept()` | 单概念展开 | Pro+web_search | 先模糊匹配（精确→子串→编辑距离<=2），未匹配则新建并展开 |
| `incremental_insert()` | 新闻增量插入 | Flash | 检查图谱存在→LLM批量定位最佳父节点→写边 |
| `get_graph_tree()` | 读取树结构 | - | 从Redis读取图谱层级树供前端渲染 |

**BFS 迭代扩展算法**：
```
queue = [(concept, depth=0) for concept in layer0]
visited = set()

while queue:
  (current, depth) = queue.pop(0)
  if current in visited or depth >= max_depth: continue
  visited.add(current)

  1. call_llm_with_tools(expand_prompt, web_search) → 发现3-8个子概念
  2. 对每个子概念：
     - 已在dynamic:concepts → 仅建边
     - 不在 → 写入 + 建边
     - 收敛判断: relevance >= 阈值 → 加入queue; 否则标记converged
  3. 匹配akshare概念池合并stocks数据
  4. 更新progress → 前端轮询
```

**并发控制**：`concept_graph:build_lock`（5分钟TTL），防止重复构建。

**全量构建前清理**：`_clear_old_graph()` 在 `build_full()` 开始时清除旧的 roots/edges/layer Sets，防止旧图谱数据残留导致 BFS 跳过新概念。

**自动补股**：`_write_concept_to_redis()` 写入概念时，若无成分股则自动触发 `find_stocks_for_concept()` 三重来源补股（API→ChromaDB→Web+LLM）。仅当最终有成分股时才加入图谱 layer Set。

**零股票过滤**：`get_graph_tree()` 构建图谱树结构时过滤掉零股票概念（不展示无成分股的概念），`/api/concepts` 概念列表接口同样过滤零股票概念。

**扩展 `dynamic:concepts`** 新增字段：`graph_depth`(int), `parent_concepts`(list), `policy_anchor`(str), `expansion_status`(str)。兼容旧数据，缺失取默认值。

## 10. 设计原则与红线

### 10.1 红线（不可违反）

| 红线 | 说明 | 原因 |
|------|------|------|
| `indicator_calc.py` 禁止 LLM 计算 | 所有 MA/量比/涨幅必须用此模块 | 数值计算必须确定性 |
| `sop_learner` 不可直接修改 system_policy | 只写 sop_pending → sop_active | 人工审核闸门 |
| SearXNG 不可在 dynamic 高频节点直接调用 | 仅供 static_graph 和 sop_learner | 限流保护 |
| 敏感值不可硬编码 | API Key/密码通过 env 注入 | 安全要求 |

### 10.2 LLM 驱动原则

所有需要"人工判断"的环节由 LLM 驱动，硬编码仅作为降级回退：

| 环节 | LLM 驱动 | 降级回退 |
|------|---------|---------|
| 政策概念词过滤 | LLM prompt 指令 | 仅过滤空 concept 字段 |
| 新闻粗筛 | LLM `is_relevant` 字段 | `is_relevant` 缺失时默认 True |
| 实体映射 | LLM `keep` 字段 | `score >= 0.6` 阈值 |
| 多因子共振 | LLM 综合判断 | 三阈值 AND（0.7/2%/2.0） |
| 技术面排序 | LLM 动态权重 | 固定权重 0.40/0.25/0.20/0.15 |
| 研报搜索 | LLM tool-call 自主搜索 | 固定模板 `f"{concept} 产业链 研报"` |
| 概念分类 | LLM 输出 confidence | 默认 0.5 |
| 概念截断 | LLM 自决数量 | 无截断 |
| 概念图谱Layer0 | LLM 从政策文本提取 | 固定搜索查询词 |
| 概念图谱扩展 | LLM+web_search发现子概念 | 固定3个子概念 |
| 概念图谱收敛 | LLM relevance_score判断 | 固定阈值0.4 |
| 概念图谱插入 | LLM 批量定位最佳父节点 | 跳过插入 |

### 10.3 降级策略

| 层级 | 降级策略 |
|------|---------|
| LLM 不可用 | 所有节点保留硬编码降级回退 |
| ChromaDB 不可用 | 语义搜索降级，不崩溃 |
| Redis 不可用 | 缓存/去重降级，不崩溃 |
| Tushare 积分不足 | 捕获异常，降级到东方财富 |
| 新闻处理异常 | 写入 `status=error`，不卡在"排队中" |

### 10.4 缓存策略

| 缓存层 | TTL | 清理策略 |
|--------|-----|---------|
| LLM 响应缓存 | 24h | Redis 自动过期 |
| URL 内容缓存 | 7d | Redis 自动过期 |
